## Распределение сокетов W5500

Чип W5500 предоставляет 8 аппаратных сокетов. Приложение использует 6 из них с настроенными размерами буферов:

| Сокет | Буфер RX/TX | Назначение |
|-------|------------|------------|
| 0 | 8 КБ / 8 КБ | MACRAW -- захват всех фреймов (LLDP, CDP, STP, ARP, статистика, мост) |
| 1 | 2 КБ / 2 КБ | DHCP-клиент (UDP порт 68) |
| 2 | 2 КБ / 2 КБ | ICMP ping и traceroute (IPRAW) |
| 3 | 2 КБ / 2 КБ | DNS (UDP 53), WoL (UDP 9), mDNS (UDP 5353), SSDP (UDP 1900), File Manager HTTP (TCP 80) |
| 4 | 1 КБ / 1 КБ | PXE TFTP listen (UDP порт 69) |
| 5 | 1 КБ / 1 КБ | PXE TFTP передача данных (эфемерный порт) |

Сокет 0 использует режим MACRAW с `MFEN=0` (MAC Filter Enable отключён) -- он принимает **все** Ethernet-фреймы на линии, а не только адресованные MAC-адресу Flipper. Это позволяет пассивным слушателям (LLDP, CDP, STP) и промискуитетному режиму моста работать.

## Потоковая модель

Приложение использует два потока:

### Основной поток (4 КБ стек)

Выполняет GUI event loop Flipper через `ViewDispatcher`. Отвечает за:

- Обработку пользовательского ввода (кнопки, навигация)
- Переключение видов (submenu, text box, byte input, text input, кастомные виды)
- Таймерные обратные вызовы (DHCP 1-секундный таймер)

Основной поток **никогда** не выполняет блокирующих сетевых операций. Все длительные задачи делегируются worker thread.

### Worker thread (8 КБ стек)

Создаётся по требованию при запуске сканирования/прослушивания/сервера. Уничтожается при завершении или отмене:

```
Пользователь выбирает функцию → основной поток создаёт worker → worker выполняет логику
                                                               → worker обновляет текстовый буфер
                                                               → основной поток рендерит UI
Пользователь жмёт Back        → основной поток: worker_running = false
                               → worker выходит из цикла
                               → основной поток join + free
```

Взаимодействие между потоками:

- **Отмена**: флаг `volatile bool worker_running`. Worker проверяет его в основном цикле и выходит при `false`.
- **Данные**: worker пишет в текстовые буферы `FuriString`, принадлежащие структуре приложения. Основной поток читает их для отображения.
- **Завершение**: worker вызывает `view_dispatcher_send_custom_event()` для уведомления основного потока.

## Модель памяти

Приложение работает со стеком 4 КБ (определён в `application.fam`). Все крупные аллокации -- в куче:

| Аллокация | Размер | Время жизни |
|-----------|--------|-------------|
| Буфер приёма фреймов (`frame_buf`) | 1600 Б | Время жизни приложения |
| Состояние графика пинга (`ping_graph`) | ~560 Б | Время жизни приложения |
| Состояние истории | ~2 КБ | Время жизни приложения |
| Текстовые буферы FuriString (20+) | Переменный | Время жизни приложения |
| Состояние моста | ~100 Б | Время жизни приложения |
| Буфер чтения файлового менеджера | 2 КБ | Во время работы файлового менеджера |

Worker thread имеет собственный стек 8 КБ, достаточный для самых глубоких цепочек вызовов парсеров протоколов.

### Поток LLDP в Auto Test (3 КБ стек)

Auto Test запускает дополнительный поток для прослушивания LLDP/CDP параллельно с основными тестами. Этот поток:

- Имеет стек 3 КБ (меньше основного worker -- парсит только LLDP/CDP TLV)
- Использует Socket 0 в режиме MACRAW для приёма LLDP/CDP-фреймов
- Выделяет собственный malloc-буфер для приёма фреймов (не использует общий `frame_buf`)
- Работает параллельно с основным worker thread, который выполняет остальные тесты Auto Test

> **Примечание о Socket 0**: в Auto Test Socket 0 разделяется между LLDP-потоком и ARP-сканированием. Эти операции выполняются **последовательно, не параллельно**: LLDP-поток завершается (или прерывается по таймауту AT LLDP wait) до начала ARP-сканирования. Одновременный доступ к MACRAW-сокету не допускается.

## Кеширование DHCP

DHCP-согласование занимает 3-15 секунд в зависимости от условий сети. Чтобы не повторять его для каждой функции, приложение кеширует первый успешный результат:

1. Первая функция, требующая сетевой конфигурации, запускает DHCP Discover/Offer
2. Результаты сохраняются в `app->dhcp_ip`, `dhcp_mask`, `dhcp_gw`, `dhcp_dns`, `dhcp_valid`
3. Все последующие функции используют кешированные значения
4. Кеш инвалидируется только при выходе из приложения

Это означает, что переключение между Ping, ARP Scan, DNS Lookup и т.д. мгновенное после первого DHCP.

## Структура данных на SD-карте

Приложение хранит данные в директории своего `appid` на SD-карте:

```
/ext/apps_data/lan_tester/
├── settings.conf           # Настройки (автосохранение, звук, DNS, пинг)
├── mac.conf                # Сохранённый MAC-адрес (случайный при первом запуске)
├── history/                # Автосохранённые результаты сканирований
│   ├── arp_20260401_143022.txt
│   ├── ping_20260401_143105.txt
│   └── ...
├── pcap/                   # Захваченный трафик в формате PCAP
│   ├── capture_20260401_150000.pcap
│   └── ...
└── pxe/                    # PXE boot-файлы
    ├── undionly.kpxe
    └── ipxe.efi
```

Пути `settings.conf` и `mac.conf` используют макрос `APP_DATA_PATH()`, который автоматически разрешается в `/ext/apps_data/lan_tester/` на основе `appid` из `application.fam`.

Директория PXE использует `EXT_PATH("apps_data/lan_tester/pxe")` как жёсткий путь, потому что TFTP-серверу нужен абсолютный путь для раздачи файлов.
